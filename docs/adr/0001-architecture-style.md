# 0001 — Architecture style: Microservices (pilot)

**Дата:** 2025-11-19  
**Контекст:** Музичний каталог (Music Player) на початковій стадії. Команда — 1-3 людини. Потрібно швидко прототипувати нові функції, зберігати можливість майбутнього масштабування.

**Рішення:** Запускати пілотний варіант мікросервісної архітектури. Починаємо з одного сервісу — `catalog-service`, який покриватиме публічний API для треків, артистів і альбомів. Решта сервісів (playlists, users) можуть зʼявитися пізніше.

**Альтернативи:** Модульний моноліт (легше розгорнути), але обмежує еволюцію окремих команд та дозволяє слабкіше розділення відповідальностей.

**Наслідки:**
- Позитив: ізоляція контекстів, простіше масштабування окремих сервісів у майбутньому.
- Негатив: початкові витрати на організацію (віддалені виклики, CI/CD, конфігурація).
- Для курсу: використовуємо mono-repo (усі сервіси в одному репозиторії) без складної інфраструктури.

**Статус:** Прийнято (pilot).  
**Умови перегляду:** Повернемось до питання розбиття, якщо:
- навантаження на `catalog` перевищить 80% CPU/IO,
- або розробка плейлистів/користувачів стане вузьким місцем для інших команд.
